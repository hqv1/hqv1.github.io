    CI/CD with Raspberry Pi

    Introduction

    My Raspberry Pi has gone through a few stages; first as a game emulator, then a media center, a Kubernetes cluster, a dust bin, and now back to a Kubernetes cluster/Development Tools Box. The setup this time went a lot more smoothly than the first time. Let's go over the steps.

    Kubernetes Cluster Setup

    I installed Ubuntu Server ARM64 onto the Raspberry Pi. "Installed" is a strong word as all it required was installing the Raspberry Pi Imager, picking the OS, flashing the SD card, and inserting it into the Pi. I choose Ubuntu Server because I'm expecting my Raspberry Pi to be a server for all sorts of applications and Ubuntu has more support than Raspbian.

    One additional benefit of using Ubuntu Server is that Canonical created a simple K8s cluster called MicroK8s. It's simple to set up but requires an OS that supports snaps.

    cgroup

    The one hiccup is cgroup is required but isn't enabled by default. This page describes the step in enabling cgroup.

    Addons

    The default version of MicroK8s is very lightweight. There are plenty of addons to spice it up. I installed dashboard, dns, helm3, and ingress. Also, fluentd, metrics-server, and prometheus sounds interesting and hopefully I can try them out later.

    External access

    The GitHub agent needs to be able to deploy my applications to the Kubernetes cluster, so I need to provide external access. The one gotcha is the Kubernete's certificate doesn't know my external IP address and clients will get warnings about possible invalid certificate when authenticating. To add the external IP address, check out this post. Make sure the IP id is greater than 100.

    GitHub Actions

    The updated GitHub Action file shows all the changes to get my CI/CD up and running.

    Wherever possible, I'm using the GitHub action instead of executing the underlying commands. For example, using Azure/k8s-deploy@v1.4 instead of executing kubectl apply .... The Actions provides setup and teardown and I wanted to learned how to use actions. All of these actions are open sourced so I can jump into the scripts if I needed to.

    Container Registry

    Instead of DigitalOcean's container registry, I'm using DockerHub. The free plan is limited to 2 private repository and 500 megs but that should be enough for this project. GitHub has their on container registry, but it's in beta and I couldn't get it enabled. There's also the option of using MicroK8's built-in registry.

    Docker Image

    The Kubernetes cluster is running on ARM64 so the container image must be built for ARM64. I also wanted an AMD64 image built in case I wanted to pull the image and run it locally on my laptop. Fortunately, there's an action for this.

    In addition, I wanted to the tag the image with the commit SHA instead of using latest. Creating and using variables isn't simple. This does require me to checkout the branch first. Otherwise I wouldn't need to checkout the branch first because build-push-action will do the checkout, build, and push for you.
